Project Name: Indoor Navigation App for Blind Users

The goal of the project is to develop a lidar-based iPhone app that assists blind individuals in navigating indoor environments. The app should provide real-time information about obstacles, potential hazards, and the general layout of the surroundings.

  1. Definitions, acronyms, and abbreviations

API (Application Programming Interface)

  • Definition: A set of rules and protocols that govern how software components interact with each other.
  • Purpose: It allows different software systems to communicate and exchange data.

AR (Augmented Reality)

  • Definition: A technology that overlays digital information, such as images or videos, onto the real world.
  • Purpose: It enhances the user's perception of reality by adding virtual elements to the physical environment.

CI/CD (Continuous Integration/Continuous Delivery)

  • Definition: A set of software development practices that automate the building, testing, and deployment of applications.
  • Purpose: It improves software quality, reduces development time, and enables faster delivery of new features.

IDE (Integrated Development Environment)

  • Definition: A software application that provides a comprehensive set of tools for software development, including a code editor, compiler, debugger, and build automation tools.
  • Purpose: It simplifies the development process and increases productivity.

KPI (Key Performance Indicator)

  • Definition: A quantifiable metric that is used to measure the performance of a business or project.
  • Purpose: It helps organizations track progress toward goals, identify areas for improvement, and make data-driven decisions.

SDK (Software Development Kit)

  • Definition: A collection of tools and libraries that developers use to create software applications for a specific platform or technology.
  • Purpose: It provides a foundation for building applications and simplifies the development process.

  1. Project organization

  1. Process model

        For this project we will use an agile approach. We will perform iterative development and frequent testing to ensure each development iteration is functioning properly and meets the project requirements. Team members will collaborate and participate in weekly sprints.

  1. Organizational structure

        The team will work together as one cohesive unit. The team leader will maintain an overview of project progress, ensuring that team objectives are met. While the team will work as one, work may be distributed across team members for efficiency.

  1. Organizational boundaries and interfaces

        Due to the structure and the size of the team, there will be a lack of distinct boundaries between team members. However, certain team members may have skills oriented towards different components of the application, meaning that they will work together more often. This application will incorporate interfaces provided by Google.

  1. Project responsibilities

        Task allocation will occur during weekly team meetings where team members will discuss what needs to be worked on during the next sprint. The team leader will divide up team members according to their own judgment, and will keep track of each member’s work and ensure accountability throughout the sprint. If a member is to fall behind, it is the team leader’s responsibility to reallocate resources to complete the tasks at hand.

  1. Managerial process

  1. Management objectives and priorities

The management objectives of this application are to deliver a reliable and safe phone app in a timely and effective manner. The priorities include high accuracy, minimal latency, and ease of use in order to keep the end-users safe at all times. To accomplish these, the team will adhere closely to the goal, keeping the users' perspectives in mind when discussing and developing.

  1. Assumptions, dependencies, and constraints

The assumption that this project will make is that the app will be able to function indoors with decent lighting. Should anything obstruct the camera’s ability to function, the user will be alerted with an auditory notification. Another assumption we will make is that the phones are capable of using either lidar features or some advanced image parsing algorithm for accurate distance data. This is essential for being able to precisely convey important information to visually impaired users.

  1. Risk management

Key risks in this project include the potential for inaccurate or varying distance sensing capabilities across devices leading to incorrect or delayed guidance. This can compromise the end user’s safety which is integral to the application’s goal. To combat this risk, the development team should take measures to rigorously test the application among different devices and handle potentially erroneous scenarios to ensure users are never in harm's way. The potential for delays brought on by unforeseen technical difficulties or reliance on outside technology is another concern. The project will incorporate a strong contingency plan, including buffer time in the timetable and frequent risk assessments, to handle this. Early in the development process, user feedback will be crucial in detecting and resolving any usability or performance concerns.

  1. Monitoring and controlling mechanisms

Agile approaches will be used in the project, including iterative development cycles and frequent sprint reviews to track work and guarantee adherence to objectives. We will regularly monitor key performance indicators (KPIs) like user satisfaction, obstacle detection accuracy, and app responsiveness. To find and fix problems as soon as possible, a specialized quality assurance team will carry out continuous testing, which will include automated unit, integration, and user acceptance tests. Stakeholders will meet on a regular basis to evaluate project milestones, talk about possible risks, and revise the project plan as needed. A feedback loop with end users will also be built to make sure the app properly satisfies their wants and expectations.

  1. Technical process

  1. Methods, tools, and techniques

Using Xcode as the integrated development environment (IDE) and Swift as the main programming language, the lidar-based iPhone app will be developed within Apple's extensive ecosystem. The application will use Apple's ARKit augmented reality framework to communicate with the lidar sensors on the iPhone, allowing for obstacle identification and real-time spatial mapping. The software will be able to understand intricate 3D environments of interior spaces through the use of ARKit, which can subsequently be examined to find potential dangers and obstacles.Lidar data will be processed and presented to users in an understandable and intuitive way using Swift's robust libraries and APIs. Xcode's simulation tools will be used for testing and debugging to make sure the app runs well on various iPhone models with lidar. This method guarantees the app's effectiveness and dependability, giving blind people a useful tool for navigating indoor environments.

  1. Software documentation

        Diagrams will be created in order to show the architectural design of the application. This will show different concepts such as data transfer, components, and any external tools such as APIs. We will also add comprehensive inline comments within the code that will allow other developers to understand the functionality of methods and lines of code. Along with that, adding detailed README files to our repositories will help developers understand the big picture function of the respective codebase. We will also add documentation for API’s, SDK, and any additional installations necessary to run the application. This includes usage examples, request/response formats, and dependencies required. Testing documentation will also be provided in order to instruct the developers on how to run unit and integration tests. This will help with analyzing bugs and any anomalies that rise in the development/production process. Lastly, a user manual will be provided in order to aid the user when navigating through our application.

  1. Project support functions

The team will have a means of communication such as holding recurring meetings on Microsoft Teams. These meetings will include daily stand-up, sprint reviews, and ticket assignments. This will help us stay on schedule for our releases and milestones. We will use GitHub in order for a smooth flow when making contributions to the codebase. We will also utilize GitHub actions to enable a CI/CD pipeline to automate testing and deployment. Also there will be quality assurance to ensure that the application is performing as expected.

  1. Work elements, schedule, and budget

The project schedule will be broken into five elements: requirements, architecture, development, testing, and deployment. Each of these phases will be executed using the Agile approach with week long sprints.

During the requirements phase, all members of the team will work together to create a requirements document detailing the requirements and plans for the application. This phase is expected to take one week’s worth of work in order to ensure a in-depth and thoughtful basis for the project

After this phase is complete the schedule for the remaining phases will be discussed during a meeting and further solidified as more details of the application are elaborated upon.

Since this is a project through the university, there is no corresponding budget. Open-source tools and SDKs will be leveraged to minimize any costs.
